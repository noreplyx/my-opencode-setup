================================================================================
 APPROACH DECISION: kind vs pattern
================================================================================

Per the ast-grep SKILL.md Three-Question Framework (Section 1):

  "pattern → when you care about the content of the code: console.log($ARG)"
  "kind → when you care about the type of AST node: all arrow_function nodes"

The task was to find ALL arrow function expressions — this is purely about node
TYPE, not about specific CODE CONTENT. Therefore, kind was the correct choice.

The YAML rule used:

    id: find-arrow-fns
    language: TypeScript
    rule:
      kind: arrow_function

Why pattern would have been WRONG (and missed matches):
  A pattern like `($$$) => $$$` requires the arrow function to have parenthesized
  parameters. It MISSES parentheseless arrow functions like:
    - n => n > 0
    - n => n * 2
  Indeed, the pattern approach found only 12 of 14 matches — missing these two.

  A pattern like `$_ => $$$` could potentially catch the no-paren case, but would
  also have other edge-case gaps and would need TWO patterns (one with parens,
  one without). kind is simpler, more correct, and covers ALL arrow functions.

 COMPARISON: kind vs pattern
================================================================================

  ------                       -------   ------

  pattern: '($$$) => $$$'       12        2   (missed n => n > 0, n => n * 2)

callbacks inside .filter() and .map(). The kind approach correctly catches them
because tree-sitter recognizes them as arrow_function AST nodes regardless of
parameter parentheses.
