=== APPROACH DECISION: pattern vs kind ===

I chose the `--pattern` approach over `--selector` (AST kind) for the following reasons:

1. `--pattern '$A => $B'` naturally matches at the `arrow_function` tree-sitter AST node.
   Verified via `--debug-query ast`, which outputs:
     arrow_function
       MetaVar $A
       =>
       MetaVar $B

2. `--selector arrow_function` is designed to *extract a sub-node* from an already-matched
   pattern node. It requires `--pattern` to be specified and paired with it. When used
   standalone (e.g., trying to match arrow_function by kind directly), it cannot work
   without a companion pattern.

3. The pattern `$A => $B` uses meta-variables ($A for parameters, $B for body) which
   naturally unify across all arrow function variants:
   - Single param without parens:  `n => n * 2`
   - Single param with parens:     `(x) => x * 2`
   - Multiple params:              `(a, b) => a + b`
   - No params:                    `() => console.log('...')`
   - Type annotations:             `(n: number) => n * 2`
   - Block body:                   `(x) => { return x; }`
   - Expression body:              `(x) => x * 2`

4. A kind-based approach (using a YAML rule with `kind: arrow_function`) would be more
   explicit and would not require meta-variable pattern matching. However, the CLI-only
   mode (without YAML config files) makes the pattern approach the most practical.

=== KEY TAKEAWAYS ===

- The pattern `$A => $B` is the simplest CLI-only approach to find ALL arrow functions.
- It matches at the `arrow_function` AST node level, covering all syntactic variants.
- `--selector` alone cannot be used without `--pattern` in CLI mode.
- A YAML rule with `kind: arrow_function` would be the "pure kind" approach but requires
  a configuration file.
- For CLI usage, the `--pattern` approach is more practical and equally correct.
