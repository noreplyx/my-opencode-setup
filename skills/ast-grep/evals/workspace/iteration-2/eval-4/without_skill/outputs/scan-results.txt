=== EXPLANATION: Why pattern over kind: arrow_function ===

I used a `pattern` approach rather than `kind: arrow_function` for the following reasons:

1. PATTERN ADVANTAGE — The pattern `const $NAME = $ARGS => $BODY` operates at the `lexical_declaration` AST
   level, meaning it naturally only matches arrow functions that are the direct value of a const/let
   variable declaration. This avoids false positives from inline arrow callbacks (e.g., callbacks inside
   .filter(), .map(), or object literals), which `kind: arrow_function` with `inside: variable_declarator`
   would incorrectly match since nested arrows are still "inside" the parent variable_declarator.

2. KIND DISADVANTAGE — Using `kind: arrow_function` as the root would require additional logic to ensure
   the arrow is a DIRECT child of a `variable_declarator` (not nested deeper inside another expression),
   which ast-grep's relationship predicates (inside, has) cannot easily express without matching through
   intermediary nodes. The `inside` predicate searches the entire ancestry chain, so an inline callback
   inside .filter() would still match if its grandparent is a variable_declarator.

3. BLOCK BODY FILTERING — Both approaches need a `not.has: kind: statement_block` to exclude block-body
   arrows. The pattern approach handles this cleanly since $BODY captures the body, and
   `not.has: kind: statement_block` filters out arrows whose body contains a block.

4. PARAMETER HANDLING — A subtlety: using `($ARGS)` (with explicit parens in the pattern) fails for
   multi-parameter arrows because `$ARGS` as a single meta-variable inside parentheses cannot span
   the comma-separated children of tree-sitter's formal_parameters node. Using `$ARGS => $BODY`
   (without parens) allows the meta-variable to capture the full parenthesized parameter list
   including any number of parameters, types, and commas.

5. CLEANER RULE — The final rule is concise:
     rule:
       pattern: const $NAME = $ARGS => $BODY
       not:
         has:
           stopBy: end
           kind: statement_block
   This is more readable and maintainable than a `kind: arrow_function` based rule with multiple
   nested relationship predicates.

In summary: pattern is preferred here because it naturally constrains the syntactic context
(const/let assignment at the top level) more precisely than kind-based rules can achieve with
simple relationship predicates.
