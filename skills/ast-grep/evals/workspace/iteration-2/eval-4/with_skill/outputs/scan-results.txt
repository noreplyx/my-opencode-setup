EXPLANATION: Why PATTERN was chosen over KIND
================================================================================

The skill's guidance (Section: "The Expanded Three-Question Framework") states:
  - Use `pattern` when you care about the *content* of the code
  - Use `kind` when you care about the *type* of AST node
  - Combine with `all: [{kind: ...}, {pattern: ...}]` to narrow a node type by
    content

In this task, we need to find:
  1. Arrow functions (this is a *type* concern)
  2. That are assigned to a `const` or `let` declaration (this is a *structural
     relationship* concern)
  3. That have a single expression body, NOT a block body (this is a *content
     shape* concern)

If we used `kind: arrow_function` alone, it would match ALL arrow functions
in the file — including inline callbacks inside .map(), .filter(), event
emitters, subscribe handlers, etc. (lines 24, 57, 61, 75-76 of the target).
We would then need complex `inside`/`has` constraints to filter to only
those assigned to a variable declaration, making the rule harder to read
and maintain.

Instead, we use `pattern`:
  - `const $NAME = $PARAMS => $BODY`
  - `let $NAME = $PARAMS => $BODY`

The `=>` token is syntactically unique to arrow functions in TypeScript —
there is no ambiguity risk. The pattern directly and readably expresses
the entire structural relationship (const/let declaration + arrow function
initializer) in a single line.

We then add a `not` clause to exclude block-body arrows (where `$BODY` would
be `{ $$$ }`), which ensures only implicit returns are matched.

Decision: PATTERN is the correct choice here because:
  1. The constraint is more about structural content (assignment pattern)
     than about the node type alone
  2. The pattern is self-documenting — it looks like the code it matches
  3. A `kind`-based approach would require extra relational rules that add
     complexity without benefit
